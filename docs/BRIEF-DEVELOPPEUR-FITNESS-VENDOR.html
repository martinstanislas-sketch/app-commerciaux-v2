0 Contexte et objectif

My Coach Ginkgo dispose d'une application web de suivi des performances commerciales qui centralise les ventes, calcule les KPIs et génère des récaps mensuels pour 3 commerciaux.

Objectif de l'intégration : Lorsqu'un commercial saisit une vente dans Fitness Vendor (plateforme Bubble.io), cette vente doit automatiquement apparaître dans l'application Suivi Commerciaux, sans aucune double saisie.

Flux de données unidirectionnel Fitness Vendor → Suivi Commerciaux (jamais l'inverse). Notre application ne va pas chercher les données, c'est Fitness Vendor qui les pousse via un appel API (webhook).

Authentification

Chaque requête doit inclure un header Authorization avec la clé API au format Bearer Token :

Authorization: Bearer wh_mycoachginkgo_2026_secret_key
Content-Type: application/json
Sécurité Cette clé doit être stockée côté serveur (jamais exposée côté client). Dans Bubble.io, utiliser l'option "Private key in header" du plugin API Connector.

Matching du commercial Le champ commercial_name doit correspondre au nom du commercial dans notre système. La comparaison est insensible à la casse ("marvin" = "Marvin" = "MARVIN"), mais l'orthographe doit être exacte. Si le nom dans Fitness Vendor diffère (ex : "Marvin Dupont"), il faudra soit le transformer côté Bubble (ne garder que le prénom), soit utiliser le système d'external_id.

Alternative : External ID

Si les noms dans Fitness Vendor ne correspondent pas exactement, on peut assigner un identifiant externe unique à chaque commercial via l'endpoint PUT /api/webhook/sales-reps/:id, puis utiliser external_id dans les ventes au lieu de commercial_name.

3 Endpoint principal : créer une vente

POST /api/webhook/sales

C'est l'endpoint que Fitness Vendor doit appeler à chaque nouvelle vente.

Payload JSON à envoyer

{
  "date":               "2026-03-03",
  "amount":             1500,
  "client_first_name":  "Jean",
  "client_last_name":   "Dupont",
  "commercial_name":    "Marvin",
  "rib_status":         "En attente"
}

Détail de chaque champ

ChampTypeRequisDescriptionContraintes
date string OUI Date de la vente Format YYYY-MM-DD strictement. Ex : "2026-03-03", jamais "03/03/2026"
amount number OUI Montant en euros Nombre positif, pas de symbole €. Ex : 1500 ou 49.90
client_first_name string non Prénom du client Texte libre. Vide si non disponible.
client_last_name string non Nom du client Texte libre. Vide si non disponible.
commercial_name string OUI* Nom du commercial "Marvin", "Magali" ou "Fabian" (insensible à la casse)
external_id string OUI* ID externe du commercial Si configuré via PUT /api/webhook/sales-reps/:id
rib_status string non Statut du RIB 3 valeurs exactes : "Reçu", "En attente", "Non fourni". Défaut = "Non fourni"
* Identification du commercial Au moins UN de ces champs est requis : commercial_name, external_id ou sales_rep_id. Nous recommandons commercial_name qui est le plus simple.

Réponse en cas de succès (201)

{
  "success": true,
  "id": 42,
  "week_start": "2026-03-02",
  "sales_rep_id": 1,
  "rib_status": "En attente"
}

Réponses d'erreur

CodeCauseExemple de body
400 Champ manquant ou format invalide {"error":"Validation échouée","details":["date est requis","amount doit être un nombre positif"]}
401 Clé API absente ou incorrecte {"error":"Non autorisé : clé API manquante ou invalide"}
403 Semaine verrouillée (saisie clôturée par le manager) {"error":"Semaine verrouillée","week_start":"2026-03-02"}

4 Mapping champs Fitness Vendor → API

Correspondance à établir dans le workflow Bubble.io entre les données de la vente Fitness Vendor et les champs de notre API :

Donnée Fitness Vendor → Champ API Transformation nécessaire
Date de la vente → date Reformater en YYYY-MM-DD. Dans Bubble : Current date:formatted as YYYY-MM-DD
Montant de l'abonnement / prix → amount Nombre seulement (pas de "€", pas de texte). Si c'est un texte, le convertir en number.
Prénom du prospect (champ prenom) → client_first_name Aucune (texte direct)
Nom du prospect (champ nom) → client_last_name Aucune (texte direct)
Nom du commercial qui a fait la vente → commercial_name Doit correspondre à "Marvin", "Magali" ou "Fabian". Si le champ contient un nom complet, extraire uniquement le prénom.
Statut du RIB (si disponible) → rib_status Mapper vers "Reçu", "En attente" ou "Non fourni". Si non disponible, ne pas envoyer le champ (défaut = "Non fourni").
Points d'attention pour le mapping
  • La date est le piège le plus courant. Bubble utilise souvent un format datetime complet. Il faut extraire uniquement YYYY-MM-DD.
  • Le montant doit être un nombre, pas une chaîne. 1500 et non "1 500 €".
  • Le nom du commercial doit matcher nos 3 noms exacts. Si Fitness Vendor utilise des noms complets ou des IDs, prévoir une conversion.
  • Les accents dans rib_status sont importants : "Reçu" avec le ç.

5 Valeurs du Statut RIB

Le champ rib_status accepte exactement ces 3 valeurs (avec accents) :

Valeur exacte à envoyerAffichageSignification
"Non fourni" Non fourni Valeur par défaut si le champ est omis
"En attente" En attente RIB demandé mais pas encore reçu
"Reçu" Reçu RIB reçu et validé

6 Guide de configuration Bubble.io

1

Installer le plugin API Connector

Aller dans l'éditeur Bubble → Plugins → Add plugins → Chercher "API Connector" → Installer.

2

Créer une nouvelle API

Dans API Connector :

  • Cliquer "Add another API"
  • Nom de l'API : MyCoachGinkgo
  • Authentication : "Private key in header"
  • Key name : Authorization
  • Key value : Bearer wh_mycoachginkgo_2026_secret_key
3

Ajouter l'appel "Envoyer Vente"

Cliquer "Add another call" :

  • Nom : Envoyer Vente
  • Use as : Action (pas "Data")
  • Data type : JSON
  • Méthode : POST
  • URL : https://VOTRE_DOMAINE/api/webhook/sales

Ajouter le header :

  • Key : Content-Type — Value : application/json

Body (cocher "Body type: JSON") :

{
  "date":              "<date>",
  "amount":            <amount>,
  "client_first_name": "<first_name>",
  "client_last_name":  "<last_name>",
  "commercial_name":   "<commercial>",
  "rib_status":        "<rib_status>"
}

Dans Bubble, les valeurs entre < > deviennent des paramètres dynamiques que vous mapperez dans le workflow.

Cliquer "Initialize call" pour valider la configuration (remplacer temporairement les paramètres par des valeurs de test).

4

Créer le Backend Workflow

Aller dans Backend workflows (onglet en haut dans l'éditeur Bubble) :

  • Trigger : Database trigger → "When a thing is created" → type = votre type de vente
  • OU : Custom event déclenché par le bouton "Valider la vente"

Ajouter une action :

  • Action type : Plugins → MyCoachGinkgo - Envoyer Vente
  • Mapper les paramètres dynamiques depuis les données de la vente créée
5

Tester l'intégration

Créer une vente test dans Fitness Vendor et vérifier :

  • Dans les logs Bubble : le call API retourne un status 201
  • Dans Suivi Commerciaux : la vente apparaît dans l'onglet Ventes
  • Le bon commercial est associé
  • Le montant et le nom client sont corrects
Resultat attendu Une fois configuré, chaque vente saisie dans Fitness Vendor apparaît automatiquement dans le tableau de bord Suivi Commerciaux en quelques secondes, avec tous les KPIs recalculés (CA, ratio, panier moyen, classements).

7 Autres endpoints disponibles

Ces endpoints sont optionnels mais peuvent être utiles :

POST /api/webhook/sales/bulk

Envoi de plusieurs ventes en un seul appel (max 100). Utile pour un import initial ou une synchronisation batch.

GET /api/webhook/sales-reps

Lister les commerciaux avec leurs IDs. Utile pour vérifier les noms exacts ou débugger un problème de matching.

PUT /api/webhook/sales-reps/:id

Assigner un external_id à un commercial. Utile si les noms dans Fitness Vendor ne correspondent pas.

La documentation détaillée de ces endpoints est disponible dans le document API-WEBHOOK-FITNESS-VENDOR.

8 Gestion des erreurs et codes HTTP

CodeSignificationAction côté Fitness Vendor
201 Vente créée avec succès Rien à faire, tout est OK
400 Données invalides Vérifier le format de la date (YYYY-MM-DD), que le montant est un nombre, et que le nom du commercial existe
401 Authentification échouée Vérifier que le header Authorization contient la bonne clé API
403 Semaine verrouillée Le manager a clôturé cette semaine. La vente ne peut plus être ajoutée pour cette période. Contacter le manager pour déverrouiller si nécessaire.
Conseil : gestion des erreurs dans Bubble Configurer une action "If API call returns error" dans le workflow pour logger les erreurs ou notifier l'administrateur. Les erreurs 400 contiennent un tableau details avec la liste précise des problèmes.

9 Tests manuels (cURL)

Pour tester l'API avant de configurer Bubble, utiliser ces commandes depuis un terminal :

curl -s -X POST https://VOTRE_DOMAINE/api/webhook/sales \

10 Checklist de mise en production

  • Serveur déployé — L'application Suivi Commerciaux est accessible via une URL HTTPS publique
  • Plugin API Connector installé dans Bubble
  • API configurée dans API Connector avec la clé d'authentification
  • Call "Envoyer Vente" configuré et initialisé avec succès
  • Backend Workflow créé avec le trigger sur création de vente
  • Mapping des champs vérifié (date format, montant number, nom commercial exact)
  • Test unitaire — créer une vente dans FV et vérifier qu'elle apparaît dans Suivi Commerciaux
  • Test des 3 commerciaux — vérifier que Marvin, Magali et Fabian sont bien identifiés
  • Test du RIB — vérifier que les 3 statuts sont correctement transmis
  • Test d'erreur — vérifier le comportement quand l'API est indisponible ou retourne une erreur
  • Monitoring — s'assurer que les erreurs de workflow sont loggées côté Bubble

11 Questions ouvertes / points à clarifier

Ces points doivent être résolus entre l'équipe My Coach Ginkgo et le développeur Fitness Vendor :

#QuestionImpact
1 Quel champ contient le montant de la vente ? L'API Fitness Vendor montre des champs prospect (nom, prénom, email...) mais pas de champ montant visible. Quel est le nom exact du champ ou de la table contenant le montant ? Mapping du champ amount
2 Comment identifier le commercial dans Fitness Vendor ? Est-ce un champ texte libre, un menu déroulant, un User Bubble ? Le format exact déterminera si on utilise commercial_name directement ou un mapping via external_id. Matching du commercial
3 Quel événement déclenche l'envoi ? Est-ce la création d'un enregistrement vente dans la base Bubble ? Un clic sur un bouton spécifique ? Un changement de statut ? Trigger du Backend Workflow
4 Le statut RIB existe-t-il dans Fitness Vendor ? Si oui, quelles sont ses valeurs actuelles ? Faut-il un mapping entre les valeurs FV et les nôtres ? Mapping du champ rib_status
5 URL de déploiement — L'application doit être déployée sur un serveur accessible publiquement. Quel hébergement sera utilisé ? Remplacement de VOTRE_DOMAINE
6 Gestion des doublons — Que faire si la même vente est envoyée deux fois (retry réseau, double clic) ? Actuellement, l'API crée un doublon. Faut-il ajouter un mécanisme de déduplication ? Logique anti-doublon à prévoir si nécessaire
